# Tailwind CSS v4 Theme Colors Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Tailwind CSS v4.3 default colors, CSS-first `@theme` configuration, bounded shared-theme loading, modern utility syntax, and custom-theme definition navigation while preserving v3 compatibility.

**Architecture:** A ranged Tailwind theme model merges an explicit v3/v4 base palette with ordered `@theme` declarations and resolves colors plus optional definition targets. The detector, hover/native provider, and navigation dispatcher consume the same model; a bounded Workspace FS loader supplies configured and relative imported theme sources with version-aware caching.

**Tech Stack:** TypeScript, Tailwind CSS 4.3 `colors` export, reactive-vscode, VS Code Workspace FS, Vitest, tsdown, pnpm.

## Global Constraints

- Use pnpm; prefix shell commands with `rtk`, except `pnpm typecheck`.
- Use RED→GREEN TDD for production behavior.
- Add `color-highlight.tailwindColorMode` (`auto | v3 | v4`), default `auto`.
- Add `color-highlight.tailwindStylesheetPaths`, default `[]`; cross-file reads require workspace trust.
- Preserve v3 colors in auto mode unless current/loaded sources contain a v4 signal.
- Use the official `tailwindcss/colors` export for v4; never execute Tailwind configs, plugins, compiler code, or arbitrary JavaScript.
- Bound theme loading to 32 files, depth 5, and 512 KiB per file using Workspace FS and open-document versions.
- Highlight and navigation must share palette selection, theme cascade, alias, reset, opacity, and utility parsing semantics.
- Preserve desktop, Web, virtual-workspace, highlight, hover, native provider, and contextual navigation behavior.
- Commit the complete implementation as `feat: support Tailwind CSS v4 themes`.
